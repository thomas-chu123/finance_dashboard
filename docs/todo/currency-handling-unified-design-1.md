# 介紹

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

本計畫旨在統一回測管理、投資組合最佳化與蒙地卡羅模擬三個模塊中的匯率處理方式。

## 目前的問題

- **回測引擎**：實作了 TWD → USD 匯率轉換（使用 `get_historical_prices("TWD=X")`）✅
- **優化引擎**：沒有匯率轉換邏輯，假設輸入數據已統一 ❌
- **蒙地卡羅模擬**：沒有匯率轉換邏輯，假設輸入數據已統一 ❌
- **結果**：混合台幣和美金資產時，計算結果不一致 ⚠️

## 解決方案

實施 **美金基準的涓滴式架構**，確保所有模塊在統一的美金基準上計算。

---

## 1. 需求與約束

### 功能需求
- **REQ-001**: 所有計算必須基於美金（USD）
- **REQ-002**: 台幣（TWD）資產在數據獲取層自動轉換為美金
- **REQ-003**: 轉換時使用真實的歷史匯率（每日 TWD=X）
- **REQ-004**: 支援多幣種的複合資產組合
- **REQ-005**: 轉換不應導致數據日期不對齊或數據遺失
- **REQ-006**: 計算結果應包含匯率轉換元數據

### 約束
- **CON-001**: 不能修改已儲存的回測結果（向後相容性）
- **CON-002**: 優化和蒙地卡羅的輸入仍允許混合幣種
- **CON-003**: 前端需要知道是否進行了匯率轉換及轉換細節

### 設計指南
- **GUD-001**: 匯率轉換邏輯應集中在市場數據層
- **GUD-002**: 每個計算引擎應提供轉換元數據
- **GUD-003**: 測試應涵蓋純美金、純台幣、混合資產三種場景

### 3.1 美金基準涓滴式架構（USD-Based Trickle Architecture）

```
┌─────────────────────────────────────────────────────┐

│  允許混合台幣和美金資產                               │
└──────────────────┬──────────────────────────────────┘

                   ▼
┌─────────────────────────────────────────────────────┐
│  路由層 (Routers)                                    │
│  ├─ backtest.py: 呼叫匯率轉換                         │
│  ├─ optimize.py: 呼叫匯率轉換                         │
│  └─ monte_carlo.py: 呼叫匯率轉換                      │
└──────────────────┬──────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────────┐
│  市場數據層 (Market Data) - 新增轉換服務              │
│  ┌──────────────────────────────────────────────┐  │

│  │   symbols: List[str],                        │  │

│  │   start_date: str,                           │  │
│  │   end_date: str                              │  │
│  │ ) -> Dict[str, Any]                          │  │
│  └──────────────────────────────────────────────┘  │
│                                                     │
│  返回：{                                             │
│    "prices_usd": {symbol: pd.Series},              │
```

### 3.2 轉換邏輯詳解

**日期對齐 ffill/bfill 策略**：
```
台灣股市交易日期:    2024-01-02, 2024-01-03, 2024-01-04 (跳過周末)
US 股市交易日期:     2024-01-01, 2024-01-02, 2024-01-03 (跳過周末)

轉換前：
- 台股行情: [101.0, 102.0, 103.0] (2024-01-02/03/04)
- 美股行情: [150.0, 151.0, 152.0] (2024-01-01/02/03)
- 匯率:     [32.5, 32.4, 32.3]   (2024-01-01/02/03, 缺 01/04)

轉換後（使用 ffill().bfill().dropna(how='all')）：
- 所有日期對齐到 2024-01-02, 2024-01-03, 2024-01-04
- 台股轉換: [101.0/32.4, 102.0/32.3, 103.0/32.3(bfill)]
- 美股轉換: [150.0(bfill), 151.0, 152.0(ffill)]
- 保留所有交易日期，沒有數據遺失
```

## 5. 複合資產計算的三個模式

### 模式 A：純美金組合  
- 例如：SPY (70%) + QQQ (30%)
- 現況：✅ 三個模組都正常
- 改進後：✅ 無變化，但會明確標記 `unconverted: ["SPY", "QQQ"]`

### 模式 B：純台幣組合  
- 例如：0050 (50%) + 0056 (50%)
- 現況：
  - 回測：✅ 正常（轉換為 USD）
  - 優化：❌ 使用原台幣價格計算（錯誤）
  - 蒙地卡羅：❌ 使用原台幣價格計算（錯誤）
- 改進後：✅ 三個模組都轉換為 USD 後計算

### 模式 C：混合組合  
- 例如：0050 (30%) + SPY (40%) + QQQ (30%)
- 現況：
  - 回測：✅ 正常（自動轉換台幣）
  - 優化：❌ 混亂（台幣美金混用）
  - 蒙地卡羅：❌ 混亂（台幣美金混用）
- 改進後：✅ 三個模組統一結果

## 6. 風險與假設

### 風險
- **RISK-001**: 匯率數據缺失時的降級策略（改用最近有效數據）
- **RISK-002**: 歷史匯率無法回溯至很久以前
- **RISK-003**: 轉換可能增加計算延遲（需性能測試）

### 假設
- **ASSUMPTION-001**: TWD=X 匯率在 yfinance 中始終可用
- **ASSUMPTION-002**: 用戶願意接受基於匯率的計算差異
- **ASSUMPTION-003**: 不需要即時匯率（接受每日收盤率）

## 7. 當前匯率計算方式

### 📍 關鍵發現：使用逐日匯率轉換

當前系統**已實現逐日匯率轉換**，而非平均匯率：

```python
# backtest_engine.py 第 113-116 行
combined = pd.DataFrame({"price": price_data[sym], "fx": twd_fx}).ffill().bfill()
price_data[sym] = combined["price"] / combined["fx"]
```

**轉換方式對比**：

| 日期 | 0050 台幣價格 | TWD=X 匯率 | 當日轉換 | 平均匯率轉換 |
|------|-------------|----------|--------|-------------|
| 2025-01-01 | ¥100.50 | 32.50 | $3.092 | $3.09 |
| 2025-01-02 | ¥100.75 | 32.45 | $3.103 | $3.10 |
| 2025-01-03 | ¥101.00 | 32.40 | $3.117 | $3.11 |

✅ **使用逐日匯率的優勢**：
- 科學準確性：還原真實歷史
- 避免套利虛值：完全重現歷史現實
- 正確的日收益率計算：已涵蓋價格漲幅 + 匯率變動
